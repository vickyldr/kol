# 数据安全与财务敏感性评估

> 给财务 / 信息安全 / 法务看的评估。一句话结论在最前面，细节在后面。

## 一句话结论

**这个工具碰敏感数据（付款金额 + 银行账号/IBAN/SWIFT/PayPal 邮箱/收款人姓名地址），
所以财务和安全一定会问「数据去哪了」。但工具本身只读、只比对、不碰钱——它是一个
风控/复核工具，反而是给财务加一道防线。唯一需要决策的敏感点只有一个：把合同/审批
PDF 识别成字段的那一步，默认会调用外部 AI（数据出公司）。这一步有三种成熟的处理方式
（企业零留存合约 / 私有化部署 / 走飞书 API），任选其一即可满足财务和安全要求。**

---

## 1. 这个工具会 / 不会做什么（先消除最大顾虑）

| | 说明 |
|---|---|
| ❌ **不发起付款** | 它不连银行、不连支付通道，不会把钱打出去 |
| ❌ **不回写财务/审批系统** | 它不修改飞书审批、不改任何账，只输出一份核对意见 |
| ❌ **不替代审批权限** | 最终批不批，仍然是人（审批人）点的。它只给「PASS/打回/转人工 + 原因」做参考 |
| ✅ **只读 + 只比对** | 读「审批单 + 合同」，把字段两两对比，算一下金额，给结论 |
| ✅ **是一道内控** | 本质是「自动复核」，专门抓人工易漏的错（金额币种不符、收款账号被改成别人、重复付款等）|

> 关键话术（可直接转述财务）：**它不是付款工具，是复核工具。不碰钱、不动账、人仍是最终决策者。
> 它能做的就是在你点「通过」之前，先帮你把合同和审批核对一遍。**

实测它已经抓出过的真实错误：币种填错（USD 写成 TRY，差约 30 倍）、**收款账号填成了另一个红人的银行账户**、
重复提交、视频数量与金额对不上等——这些正是财务最怕漏掉的。

---

## 2. 它碰哪些敏感数据

- **财务数据**：付款金额、币种、单价。
- **收款信息（属个人/企业敏感信息）**：银行账号、IBAN、SWIFT、PayPal/Payoneer 邮箱、
  收款人姓名、收款地址、邮编。
- **个人信息（PII）**：KOL / 收款人的姓名、邮箱、地址、银行账户。
  - 若涉及欧盟红人，触及 **GDPR**；国内适用 **个人信息保护法（PIPL）**。
  - 建议按「数据最小化 + 限定留存 + 访问控制」处理（见第 4 节）。

---

## 3. 数据流向：拆成两块看，敏感点只有一个

整套系统分两部分，**敏感度完全不同**，财务/安全只需盯住第二块：

### 第①块：比对引擎（rules.py / models.py / dedup.py）—— 完全安全

- 纯本地 Python 代码，**零外部网络调用**。
- 做的全是「字段相等吗、金额除得尽吗、有没有重复」这类确定性运算。
- 给定相同输入永远得到相同结果，**可审计、可复现、可单测**（已有 44 个测试用例）。
- **数据完全不出公司这一步。**

### 第②块：PDF→字段的「AI 识别」（extract.py）—— 唯一敏感点 ⚠️

- 合同是 PDF、审批可能是截图，要先「读成字段」才能比对。
- 默认实现是调用 **外部大模型 API（Anthropic Claude）** 来识别。
- **这一步会把合同/审批的内容（含银行账号、IBAN、金额等）发送到公司外部。**
- **这就是财务/安全唯一真正需要拍板的地方。**

> 注意：如果将来走**飞书 API** 直连（见 README），审批数据本来就是结构化的，
> 审批侧根本不需要过 AI；只有合同 PDF 需要识别——出境数据量大幅减少。

---

## 4. 给技术团队的三种合规落地方案（任选其一即可过审）

针对第②块那个敏感点，按「改动成本」从低到高：

### 方案 A：用企业版 / 零数据留存（最快）
- Anthropic（及主流云厂商）面向企业提供 **零留存（Zero Data Retention）、不用于模型训练**
  的合约条款，可签 **DPA（数据处理协议）**。
- 数据虽然过 API，但有合同约束「不留存、不训练、用完即弃」。
- 适合：想最快上线、法务能接受「有 DPA 的境外/云 API」。

### 方案 B：私有化 / 内网部署（最稳，数据不出内网）
- 字段识别改用 **公司内部 OCR + 私有/自托管模型**（或国内合规云的私有化部署）。
- 比对引擎本来就是本地的，这样**全链路数据都留在内网**。
- 适合：财务/安全要求「敏感数据绝不出公司」。
- 代价：要部署/采购一个 OCR 或私有模型；比对逻辑零改动（接口已抽象）。

### 方案 C：走飞书 API + 只识别合同（数据最小化）
- 拿到飞书管理员授权后，审批字段直接从飞书结构化读取，**审批侧不过 AI**。
- 只有合同 PDF 需要识别，可叠加方案 A 或 B。
- 出境/外发数据量最小，且数据停留在公司已有的飞书生态内。

> 三种可组合。例如「飞书取审批（C）+ 合同用私有 OCR（B）」= 全程不出内网。

---

## 5. 数据留存（这一点对财务/安全很友好）

- 工具**默认不保存**合同/审批原文。
- 唯一持久化的是**重复提交检测的「指纹」**：把 `项目+KOL+账户+金额+币种` 算成一个
  **sha256 哈希**存本地 JSON。哈希**不可逆**、**不含明文**，只用来判断「这单是不是处理过」。
- 可配置定期清理；也可改为存到公司内部数据库并纳入现有数据治理。

---

## 6. 访问与审计建议（给安全团队）

- 部署在**公司内网/受控服务器**，按现有权限体系限制谁能跑、谁能看结果。
- API Key / 飞书 App Secret 等放入**密钥管理**（环境变量/密钥库），不写进代码、不进 Git。
- 比对引擎确定性 + 已有测试，**每条结论都能回溯到是哪条规则、对比了哪两个值**——
  天然适合审计。

---

## 7. 给财务的一页话术（可直接复制转述）

> 这个工具是**付款审批的自动复核**，不是付款工具：
> 1. **不碰钱、不动账、不替代审批**——只读「审批单+合同」，比对后给「通过/打回/转人工 + 原因」，人仍是最终决策者。
> 2. **算账和判断全在本地代码**，确定性、可审计、可复现，数据不出这一步。
> 3. **唯一对外的是「PDF 识别成字段」那一步**，可改成 ①企业零留存合约 ②私有化内网部署 ③走飞书 API，把数据留在内网/受控范围。
> 4. **本地只存不可逆指纹做查重，不存原文。**
> 5. 它的作用是**加一道防线**：自动抓出币种填错、收款账号被改成别人、重复付款这类人工易漏的错。

---

## 8. 结论与建议

- **敏感性**：中—高（碰钱的金额 + 银行账号/PII），但**风险集中在单一环节**（PDF→字段识别）。
- **可控性**：高。比对核心本地化、确定性、可审计；敏感环节有三套成熟缓解方案。
- **建议**：先用方案 A（带 DPA 的企业零留存）快速试点验证业务价值，
  同时让技术评估方案 B/C 做长期合规落地。
