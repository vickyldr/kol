# KOL 付款审批自动核对 —— 交接说明（给领导与技术团队）

## 这是什么

一个**付款审批自动复核工具**。把业务每天手动做的「KOL 付款审批 ↔ 合同」核对自动化：
读取审批单和所附合同，逐字段比对、算账、查重，输出 **通过 / 打回 / 转人工 + 具体原因**。

> 现状：核心**核对引擎已完成并跑通**（19 条规则、44 个单元测试全过），
> 已用一线真实数据连续校准多日，抓出过币种填错、收款账号张冠李戴、重复提交等真实错误。
> 缺的是「数据怎么自动进来」的对接（飞书 API 或私有化 OCR），属工程对接，逻辑零改动。

## 为什么值得做

- 业务现在**纯人工**核对，量大、易漏，错一笔就是真金白银打错/多付。
- 这工具把「金额÷单价、币种、收款账号、重复付款」这类机械核对自动化，
  人只需看它标「转人工」的少数单子。**是给财务加一道内控防线，不是替代审批。**

## 核心设计（一句话）

**算术和判断用写死的代码做，AI 只负责把图文「读成字段」。**
金额÷单价、数量比对、PASS/FAIL 全在 `rules.py` 里，确定性、可复现、可单测——
绝不让 AI 估金额。这是可靠性的根本保证。

## 19 条核对规则

见 `README.md` 顶部表格（项目/KOL/账户名/币种/金额勾稽/收款账号 IBAN·SWIFT·邮箱/
合同乙方名称/上线类型/纯 YouTube 长视频/重复提交等）。
结论三档：**PASS 通过 / FAIL 打回 / FLAG 转人工**。

## 代码结构

```
src/kol_audit/
  models.py    审批 / 合同 的数据模型（字段契约，所有数据最终落到这两个对象）
  rules.py     ★ 19 条核对规则，纯函数、可单测——核对逻辑全在这，最值得读
  dedup.py     重复提交检测（本地存不可逆指纹）
  extract.py   用 AI 从 PDF/截图提取字段（← 唯一对外的环节，见数据安全评估）
  batch.py     批量核对 + 自动配对（一次核多单）
  report.py    生成报告
  cli.py       命令行入口
feishu/
  client.py    飞书审批 API 接入骨架（拿到管理员权限后补全 TODO 即变全自动）
tests/
  test_rules.py  44 个规则单测
```

## 上线还差什么（按优先级）

1. **数据入口对接**（工程量主要在这）。二选一或都做：
   - **飞书 API 直连**：需**管理员**在飞书开放平台建「企业自建应用」，开通审批读取 +
     附件下载权限，拿到 App ID/Secret。`feishu/client.py` 已搭好骨架，补 TODO 即可。
   - **私有化 OCR/识别**：把 `extract.py` 的识别后端换成公司内部 OCR/私有模型（接口已抽象）。
2. **合规落地**：见 `数据安全与财务敏感性评估.md`——确定 PDF 识别那步走哪种方案
   （企业零留存 / 私有化 / 飞书 API）。**这步要先跟财务和安全对齐。**
3. **部署**：内网/受控服务器，定时任务跑，结果回推飞书或汇总到看板。密钥进密钥库。

## 重点提醒技术

- **核对逻辑（rules.py）是资产，别动它的判断**——它是用真实业务数据一条条校准出来的，
  每条规则背后都有真实踩过的坑。要改先读规则注释和对应测试。
- **数据入口可替换**：手动喂 / 截图识别 / 飞书 API 自动抓，核对逻辑零改动。
- **敏感数据只在 `extract.py` 那步出公司**，其余全本地。合规重点盯那一处即可。
