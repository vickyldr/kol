# 日常使用：手动文件模式（不需要 API / 不需要装软件）

适用于：拿不到飞书 API、也不想装软件部署服务器的情况。
直接把这个 Claude Code 项目当核对工具用。

## 你每天怎么做

1. **从飞书导出两个文件**
   - 合同：飞书里打开合同附件 → 分享/存到 iPhone 的「文件」App
   - 审批：飞书里把审批下载 → 也存到「文件」
   （也可以直接发审批/合同的**截图**，效果一样；只是金额这类数字，
   截图请尽量清楚，或在消息里顺手打字确认一下金额。）

2. **传给 Claude**
   - 打开本项目 → 输入框 ➕ → 选这两个文件 → 发一句「核一下」

3. **看结论**
   - Claude 会读文件、跑核对引擎，给出 PASS / FAIL + 具体原因

## Claude（助手）收到文件后要做的事

> 这段是给助手看的操作规范，保证每次结果一致、可复现。

1. 用 Read 工具读「合同」和「审批」两个文件，提取字段，分别构造成
   `tests/fixtures` 那种结构的 Approval / Contract JSON。
   - **金额、单价、数量这类数字要逐字确认**；任何一个读不准的数字，
     **必须停下来让用户确认**，不许猜——读错一个小数点结论就反了。
2. 把两个 JSON 存成临时文件，调用真正的引擎出结论（不要靠脑算）：
   ```bash
   PYTHONPATH=src python -m kol_audit.cli \
     --approval-json /tmp/approval.json \
     --contract-json /tmp/contract.json --no-record
   ```
3. 把引擎输出的报告原样给用户，再用一句白话总结结论。
4. 重复提交检测：如果希望跨天/跨单生效，去掉 `--no-record`，
   并让 `processed_approvals.json` 持久保存（注意本项目是临时环境，
   要长期记录需另存或提交到仓库）。

## 已知的两个人工确认点（来自真实样例）

- **项目名**：有的合同正文里不印项目代号（例如审批写 Ophiucus，
  英文合同里没这个词），这条没法自动对，会标出来请人工确认。
- **金额**：整页长截图会把数字缩得很小，容易看不清小数点。
  优先用导出的文件，或用户口头确认金额。
