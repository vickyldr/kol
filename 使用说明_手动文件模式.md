# 日常使用：手动文件模式（不需要 API / 不需要装软件）

适用于：拿不到飞书 API、也不想装软件部署服务器的情况。
直接把这个 Claude Code 项目当核对工具用。

## 你每天怎么做

1. **从飞书导出两个文件**
   - 合同：飞书里打开合同附件 → 分享/存到 iPhone 的「文件」App
   - 审批：飞书里把审批下载 → 也存到「文件」
   （也可以直接发审批/合同的**截图**，效果一样；只是金额这类数字，
   截图请尽量清楚，或在消息里顺手打字确认一下金额。）

2. **传给 Claude**
   - 打开本项目 → 输入框 ➕ → 选这两个文件 → 发一句「核一下」

3. **看结论**
   - Claude 会读文件、跑核对引擎，给出 PASS / FAIL + 具体原因

## Claude（助手）收到文件后要做的事

> 这段是给助手看的操作规范，保证每次结果一致、可复现。

1. 用 Read 工具读「合同」和「审批」两个文件，提取字段，分别构造成
   `tests/fixtures` 那种结构的 Approval / Contract JSON。
   - **金额、单价、数量这类数字要逐字确认**；任何一个读不准的数字，
     **必须停下来让用户确认**，不许猜——读错一个小数点结论就反了。
2. 把两个 JSON 存成临时文件，调用真正的引擎出结论（不要靠脑算）：
   ```bash
   PYTHONPATH=src python -m kol_audit.cli \
     --approval-json /tmp/approval.json \
     --contract-json /tmp/contract.json --no-record
   ```
3. 把引擎输出的报告原样给用户，再用一句白话总结结论。
4. 重复提交检测：如果希望跨天/跨单生效，去掉 `--no-record`，
   并让 `processed_approvals.json` 持久保存（注意本项目是临时环境，
   要长期记录需另存或提交到仓库）。

## 项目核对的正确口径（来自真实样例，已确认）

- **审批的「项目」要和合同 WHEREAS 条款里写的「要推广的 App 名」完全一致**
  （例如合同写 VivaVideo，审批项目就该是 VivaVideo）。
  - 提取合同时，`project` 字段填的是**那个 App 名**，不是甲方公司名（Quvideo）。
  - 对不上 → **打回**（真实案例：审批项目 Rythmix、合同却是 VivaVideo，应打回）。
  - 合同确实没写 App 名 → 标「人工确认」，不误判失败。

## 预付款 / 分期 / 尾款的识别

- 不能只看「审批流程」字段。有的单流程写「现付款流程」，但**备注里**才点明
  是预付尾款（如「合同为预付款、已预支付50%、现支付剩余50%」）。
- 只要审批流程是预付款流程，**或**备注出现「预付款/已预支付/分期/尾款/支付剩余」，
  就当 `is_prepayment=True`：单独标记转人工，跳过「金额÷单价」视频数量核对。

## 金额读取提醒

- 整页长截图会把数字缩得很小，容易看不清小数点。
  **优先用导出的 PDF 文件**（数字清晰），或让用户口头确认金额。

## 用户不需要分辨/配对文件

- 合同文件名常是一串乱码（camdy.cn 的哈希），用户分不清哪个是哪个——**没关系**。
- 助手负责读每个文件的内容、认出是谁（KOL/账户名）、自动把审批和合同配对。
- 用户一次只能传 5 个文件也没关系：**分多条消息发，助手攒着一起核**；
  暂时没配上对的单子，标「找不到对应合同」，等它的另一半传上来再核。

## 收款信息核对（按审批实际填写的字段）

审批付款栏会填的收款字段（不同国家/收款方式填的不一样）：
账户名、收款方式、PayPal/Payoneer 邮箱、IBAN、SWIFT、银行账号、国家、地址、邮编。
- **收款账号**（邮箱 / IBAN / SWIFT / 银行账号）→ 与合同不一致**直接打回**（钱流向的关键）。
- **收款辅助信息**（国家 / 地址 / 邮编）→ 不一致**标人工确认**（写法常有差异，不硬打回）。
- 以审批实际填了的字段为准，去合同里找对应项比对；两边都有才比。
